TensorFlow vs PyTorch

Both are open-source deep learning frameworks, but they differ fundamentally in how they build and execute computation graphs. Understanding the difference shapes how you think about debugging, research, and production deployment.

Feature TensorFlow 1.x / Static PyTorch / Dynamic
Graph execution Static (define-then-run). Build graph first, then feed data via Session.run(). Dynamic (define-by-run). Graph is built as code executes — just write Python.
Debugging Hard — errors appear at graph execution time, not at the line that caused them. Easy — standard Python debugger, print(), pdb all work normally.
Control flow Special ops: tf.cond, tf.while_loop Native Python: if, for, while
Research friendliness Verbose; iterating quickly on new architectures is slow. Preferred in research — write architecture changes like normal code.
Production / deployment Strong ecosystem: TensorFlow Serving, TFLite, TFX. TorchScript, ONNX export, TorchServe — closing the gap rapidly.
Ecosystem dominance Still used in industry, especially at Google. Dominant in academia; growing fast in industry. Most new papers use PyTorch.
NumPy similarity Moderate — TF2 with eager mode is closer. Very high — PyTorch API closely mirrors NumPy.
Key insight: TensorFlow 1.x treated your Python code as a description language for a graph that ran separately. PyTorch treats Python itself as the runtime — your code is the computation.

The Static vs Dynamic Graph Analogy

Think of static graphs like a compiled language (C++): you declare everything upfront, then run. Dynamic graphs are like an interpreted language (Python): each line executes immediately. Dynamic graphs are slower to optimize but vastly easier to work with.

TensorFlow 1.x (static graph)legacy style
import tensorflow as tf

# Step 1: BUILD the graph (nothing runs yet)
x = tf.placeholder(tf.float32, shape=[None, 784])
W = tf.Variable(tf.zeros([784, 10]))
y = tf.matmul(x, W)

# Step 2: RUN the graph in a session
with tf.Session() as sess:
    sess.run(tf.global_variables_initializer())
    result = sess.run(y, feed_dict={x: my_data})
# Debugging? Good luck — errors point to sess.run(), not the line above.
PyTorch (dynamic graph)modern style
import torch

# Everything runs immediately, line by line
x = torch.zeros(1, 784)
W = torch.zeros(784, 10)
y = x @ W        # matmul — runs NOW
print(y.shape)   # torch.Size([1, 10]) — inspect anytime

# Use a normal Python if-statement inside your model
if y.sum() > 0:
    y = y * 2

Tensors

A tensor is a multi-dimensional array — the fundamental data structure in PyTorch. Everything from a single number to a batch of images is a tensor.

Linear Algebra in PyTorch

Neural networks are linear algebra. Understanding these operations at the tensor level demystifies what happens inside every layer.

Element-wise vs Matrix Operations

Python · PyTorchbasic operations
A = torch.tensor([[1.,2.],[3.,4.]])
B = torch.tensor([[5.,6.],[7.,8.]])

# Element-wise (shapes must match)
A + B              # [[6,8],[10,12]]
A * B              # [[5,12],[21,32]]  — NOT matrix multiply!
A ** 2            # [[1,4],[9,16]]
torch.sqrt(A)    # element-wise sqrt

# Matrix multiplication  ← the important one
A @ B              # [[19,22],[43,50]]
torch.matmul(A, B) # same thing
torch.mm(A, B)     # 2D only version

# Batched matmul (e.g., batch of weight matrices)
X = torch.randn(32, 10, 5)  # batch of 32 matrices
W = torch.randn(32,  5, 8)
out = X @ W              # [32, 10, 8]  — batched!

Transpose and Dot Product

Python · PyTorchtranspose · dot product
A = torch.randn(3, 4)

# Transpose
A.T               # shape [4, 3]
A.transpose(0, 1) # same — swap dims 0 and 1
A.permute(1, 0)   # same for 2D; powerful for N-D tensors

# Dot product (1D vectors)
u = torch.tensor([1., 2., 3.])
v = torch.tensor([4., 5., 6.])
torch.dot(u, v)   # 1*4 + 2*5 + 3*6 = 32.0

# For 2D: use @ or matmul
# Linear layer: y = x @ W.T + b
x = torch.randn(1, 4)    # 1 sample, 4 features
W = torch.randn(3, 4)    # 3 output neurons, 4 inputs
b = torch.randn(3)
y = x @ W.T + b          # [1, 3] — one prediction per neuron

Reductions and Norms

Python · PyTorchreductions · norms
X = torch.randn(4, 3)

# Reductions over all elements
X.sum()         # scalar
X.mean()        # scalar
X.max()         # scalar max value

# Reductions along a dimension
X.sum(dim=0)    # sum over rows → shape [3]
X.sum(dim=1)    # sum over cols → shape [4]
X.mean(dim=0, keepdim=True)  # shape [1, 3] — keeps dim

# Argmax (for classification)
logits = torch.tensor([0.1, 2.5, 0.8])
logits.argmax()   # tensor(1) — predicted class

# Norms
torch.norm(X)              # Frobenius norm
torch.norm(X, dim=1)       # L2 norm per row
torch.linalg.norm(X, ord=1)  # L1 norm

# Useful in deep learning: normalize embeddings
X_norm = X / X.norm(dim=1, keepdim=True)

Broadcasting

Broadcasting allows operations between tensors of different shapes — PyTorch expands the smaller tensor automatically, without copying memory.

Autograd — Automatic Differentiation

PyTorch's autograd engine automatically computes gradients. Every tensor operation is recorded in a computation graph; calling .backward() propagates gradients back through it via the chain rule.

Why this matters: You never hand-derive gradients. Write your forward pass — PyTorch handles the backward pass automatically.
Python · PyTorchautograd basics
import torch

# requires_grad=True tells PyTorch to track this tensor
x = torch.tensor(3.0, requires_grad=True)

# Forward pass: build computation graph
y = x ** 2 + 2 * x + 1   # y = x² + 2x + 1

# Backward pass: compute dy/dx
y.backward()

print(x.grad)   # tensor(8.) — dy/dx = 2x+2 = 2(3)+2 = 8 ✓

# Gradients accumulate — zero them before next backward!
x.grad.zero_()   # in-place zero (trailing _ = in-place in PyTorch)
Python · PyTorchautograd with tensors (the real use case)
# Typical setup: learnable weights
W = torch.randn(3, 4, requires_grad=True)
b = torch.zeros(3,    requires_grad=True)

x = torch.randn(4)    # input — no grad needed

# Forward
y_pred = W @ x + b     # [3]
loss   = (y_pred ** 2).sum()

# Backward
loss.backward()

print(W.grad.shape)   # [3, 4] — same shape as W
print(b.grad.shape)   # [3]

# Detach from graph when you don't need grads
y_np = y_pred.detach().numpy()  # safe to convert

# Context manager to disable grad tracking (e.g. inference)
with torch.no_grad():
    y_pred_eval = W @ x + b   # no graph built, faster

nn.Module — Building Neural Networks

torch.nn.Module is the base class for all neural network components. Any network, layer, or block you create should inherit from it.

Python · PyTorchanatomy of an nn.Module
import torch
import torch.nn as nn

class SimpleNet(nn.Module):
    def __init__(self, in_features, hidden, out_features):
        super().__init__()           # always call this first!

        # Define learnable layers as attributes
        self.fc1 = nn.Linear(in_features, hidden)
        self.fc2 = nn.Linear(hidden, out_features)
        self.relu = nn.ReLU()

    def forward(self, x):
        # Define the forward pass
        x = self.fc1(x)       # linear transform
        x = self.relu(x)      # activation
        x = self.fc2(x)       # output layer
        return x

# Instantiate and inspect
model = SimpleNet(784, 256, 10)
print(model)
# SimpleNet(
#   (fc1): Linear(in_features=784, out_features=256, bias=True)
#   (fc2): Linear(in_features=256, out_features=10, bias=True)
#   (relu): ReLU()
# )

# Count parameters
total = sum(p.numel() for p in model.parameters())
print(f"Parameters: {total:,}")   # 203,530

Python · PyTorchtorch.nn layers

# Convolutional

Activation Functions

Activation functions introduce non-linearity into the network. Without them, stacking linear layers is equivalent to a single linear layer.

▶ Activation Function Explorer

ReLU(x) = max(0, x) — simple, fast, most widely used in hidden layers.

Python · PyTorchactivation functions
import torch
import torch.nn as nn
import torch.nn.functional as F

x = torch.randn(5)

# As nn.Module (use in Sequential or as attribute)
nn.ReLU()(x)         # max(0, x)
nn.Sigmoid()(x)      # 1 / (1 + e^-x)   → output in (0,1)
nn.Tanh()(x)         # (e^x - e^-x)/(e^x + e^-x) → (-1,1)
nn.GELU()(x)         # used in Transformers (BERT, GPT)
nn.LeakyReLU(0.01)(x) # fixes dying ReLU: slope 0.01 for x<0

# As functional (use inside forward())
F.relu(x)
F.sigmoid(x)
F.tanh(x)
F.gelu(x)

# Softmax — for output layer in multi-class classification
logits = torch.tensor([1.0, 2.0, 0.5])
probs  = F.softmax(logits, dim=0)
# tensor([0.2119, 0.5761, 0.2120]) — sums to 1.0

# Note: nn.CrossEntropyLoss applies softmax internally
# — do NOT add Softmax at the end when using it!

11. What is the "dying ReLU" problem?

A
ReLU outputs explode to infinity for large positive inputs
B
Neurons get stuck outputting 0 permanently because large negative inputs give zero gradient
C

Loss Functions & Optimizers

The loss function measures how wrong the model is. The optimizer uses gradients to update weights to reduce that loss.

Common Loss Functions

Python · PyTorchloss functions
import torch.nn as nn

# Regression
nn.MSELoss()              # Mean Squared Error: mean((y_pred - y)²)
nn.L1Loss()               # Mean Absolute Error: mean(|y_pred - y|)
nn.HuberLoss(delta=1.0)   # robust: L2 near 0, L1 for outliers

# Binary classification
nn.BCEWithLogitsLoss()     # raw logits → sigmoid → BCE (preferred)
nn.BCELoss()              # if you already applied sigmoid

# Multi-class classification
nn.CrossEntropyLoss()      # raw logits → softmax → NLL (preferred)
nn.NLLLoss()              # if you already applied log_softmax

# Example usage
criterion = nn.CrossEntropyLoss()
logits = torch.randn(32, 10)   # [batch, num_classes]
labels = torch.randint(0, 10, (32,))  # [batch] integer labels
loss = criterion(logits, labels)
print(loss.item())   # scalar Python float

Python · PyTorchoptimizers
import torch.optim as optim

The Training Loop

The training loop is the core engine of supervised learning. Every epoch it repeats the same four steps: forward → loss → backward → update.

Python · PyTorchcomplete training loop template
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset

# ── Setup ──────────────────────────────────────
device    = "cuda" if torch.cuda.is_available() else "cpu"
model     = SimpleNet(784, 256, 10).to(device)
criterion = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=1e-3)

# ── Data ───────────────────────────────────────
X_train = torch.randn(1000, 784)
y_train = torch.randint(0, 10, (1000,))
dataset = TensorDataset(X_train, y_train)
loader  = DataLoader(dataset, batch_size=32, shuffle=True)

# ── Training loop ──────────────────────────────
for epoch in range(20):
    model.train()
    running_loss = 0.0

    for X_batch, y_batch in loader:
        X_batch = X_batch.to(device)
        y_batch = y_batch.to(device)

        # 1. Zero gradients from previous step
        optimizer.zero_grad()

        # 2. Forward pass
        logits = model(X_batch)          # [32, 10]

        # 3. Compute loss
        loss = criterion(logits, y_batch)

        # 4. Backward pass — compute gradients
        loss.backward()

        # 5. Update weights
        optimizer.step()

        running_loss += loss.item()

    avg_loss = running_loss / len(loader)

    # ── Validation ─────────────────────────────────
    model.eval()
    correct = 0
    with torch.no_grad():
        for X_val, y_val in val_loader:
            preds   = model(X_val.to(device))
            correct += (preds.argmax(dim=1) == y_val.to(device)).sum().item()
    acc = correct / len(val_loader.dataset)
    print(f"Epoch {epoch+1}: loss={avg_loss:.4f}  val_acc={acc:.3f}")

14. What is the correct order of operations in a single training step?

A
backward → forward → zero_grad → step
B
zero_grad → forward → loss → backward → step
C
forward → backward → zero_grad → step
D

The standard order is: (1) zero_grad — clear old gradients, (2) forward — compute predictions, (3) loss — measure error, (4) backward — compute new gradients, (5) step — update weights. Zeroing must come before backward, not after.

Shuffling prevents the model from learning spurious patterns from sample ordering. Mini-batching (batch_size=32) gives noisy but computationally efficient gradient estimates compared to full-batch gradient descent.
